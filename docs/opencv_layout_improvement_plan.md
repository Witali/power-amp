# План улучшения OpenCV-распознавания страниц

Этот документ фиксирует направления развития детектора макета страниц журнала. Цель - точнее различать текст, схемы, диаграммы, фотографии, таблицы и служебные поля перед OCR.

## Цели

- Уменьшить путаницу между `schematic`, `diagram`, `table` и плотным текстом.
- Не допускать пересекающихся областей: спорные участки должны принадлежать только одному блоку.
- Стабильно находить маленькие схемы, подписи рисунков и вертикальные боковые надписи.
- Использовать исправленные вручную страницы как проверочный набор.
- Сохранять Python-пайплайн как эталон, параллельно развивая native Go/C-прототипы.

## Приоритет 1: признаки текста

- Считать статистику connected components: медианная высота букв, разброс размеров, плотность компонентов на строку.
- Определять регулярность строк: расстояние между базовыми линиями, высота строк, интервалы между колонками.
- Отдельно классифицировать заголовки: крупные жирные буквы не должны считаться схемой из-за сильных вертикальных/горизонтальных контуров.
- Лучше поддерживать вертикальный и повернутый текст в боковых полях.

## Приоритет 2: признаки схем

- Развить `component_signature_score`: резисторы, конденсаторы, диоды, транзисторы, катушки, трансформаторы, микросхемы.
- Искать позиционные обозначения около line-art: `R`, `C`, `VD`, `VT`, `DD`, `L`, `T`, `HL`, `FU`, `SA`.
- Строить граф проводников: горизонтальные/вертикальные сегменты, T-соединения, узлы, длинные связи между компонентами.
- Отличать схему от диаграммы по числу соединений и типовых УГО, а не только по плотности линий.

## Приоритет 3: признаки диаграмм

- Искать оси, шкалы, повторяющиеся ступенчатые/импульсные линии.
- Объединять несколько близких осциллограмм в один блок `diagram`.
- Не повышать диаграмму до `schematic`, если есть много горизонтальных графиков, но мало УГО радиодеталей.

## Приоритет 4: подписи и связи блоков

- Надежно находить `Рис.`, `Рисунок`, `Fig.`, `Figure` и номер рисунка.
- Хранить связь подписи с ближайшей схемой, диаграммой или фотографией.
- OCR подписей делать отдельно от OCR основного текста.
- Если подпись лежит внутри visual-блока, подсвечивать ее, но не дробить весь блок без необходимости.

## Приоритет 5: ownership и отсутствие пересечений

- Развить текущий слой `resolve_block_overlaps`.
- Для каждого пересечения классифицировать спорный участок отдельно.
- Спорную область оставлять у наиболее вероятного владельца, а из второго блока вырезать через `outline`.
- Проверять, что после постобработки реальные маски блоков не пересекаются.

## Приоритет 6: multi-scale анализ

- Запускать morphology на нескольких масштабах: мелкий текст, обычный текст, заголовки, line-art.
- Для каждого масштаба сохранять candidate boxes и затем объединять их через scoring.
- Использовать более мягкие kernel-ы для старых сканов с неравномерной печатью.

## Приоритет 7: OCR feedback loop

- Для блоков `text` запускать быстрый OCR-пробник.
- Если OCR дает много русских слов и высокую уверенность, укреплять классификацию `text`.
- Если OCR слабый, а line-art/component score высокий, пересматривать блок как `schematic` или `diagram`.
- Проверять распознанный текст spell-checker-ом и OCR-эвристиками.

## Приоритет 8: калибровка на проверенных страницах

- Сохранять ручные исправления как ground truth.
- Для каждой проверенной страницы хранить исходник, preview, `layout.json` и заметки о проблемных местах.
- Автоматически считать confusion matrix по классам.
- Перед изменением порогов прогонять регрессионный набор страниц.

## Приоритет 9: native-реализация

- Продолжать прототип `native/go-layout` как переносимый CLI без внешних зависимостей.
- Сначала переносить независимые части: thresholding, connected components, histograms, signatures.
- Затем сравнивать JSON-вывод Go и Python на одинаковых страницах.
- Python-скрипты не удалять: они остаются более гибким исследовательским эталоном.

## Проверки после изменений

- `python -m unittest discover -s tests`
- `git diff --check`
- Визуально проверить PNG-превью на нескольких страницах:
  - `b.2000-02.036`
  - `b.2000-02.037`
  - страницы с фотографиями, схемами и вертикальными боковыми полями.
- Проверить, что маленькие схемы не пропадают, а диаграммы не превращаются в схемы без component signatures.

## Ближайшие практические шаги

1. Добавить метрику connected-component statistics для текста.
2. Добавить junction graph для схем.
3. Расширить component signatures на катушки, трансформаторы и микросхемы.
4. Сохранить несколько вручную проверенных страниц как ground truth.
5. Сделать команду сравнения `Python layout` vs `Go layout`.
